Introduction

A linear mixed model can be written as

\[ \boldsymbol{y} = \boldsymbol{X}\boldsymbol{\beta} + \boldsymbol{Z}\boldsymbol{u} + \boldsymbol{\epsilon} \]

where \(\boldsymbol{X}\) and \(\boldsymbol{Z}\) are the two design matrices we need to somehow construct when dealing with this type of models. \(\boldsymbol{X}\) is the design matrix for the common (a.k.a. fixed) effects, and \(\boldsymbol{Z}\) is the design matrix for the group-specific (a.k.a. random or varying) effects.

It is quite easy to obtain the design matrix \(\boldsymbol{X}\) in R using its popular formula interface. In Python, patsy provides an equivalent functionality. Unfortunately, there arenât as many alternatives to compute the matrix \(\boldsymbol{Z}\).

In R, thereâs lme4, the statistical package par excellence for mixed models. It extends the base formula interface to include group-specific effects via the pipe (|) operator and internally computes both \(\boldsymbol{X}\) and \(\boldsymbol{Z}\) without the user noticing. Thatâs great!

In Python, we are working in formulae, a library to handle mixed model formulas in Bambi. In this process, Iâve found Fitting Linear Mixed-Effects Models Using lme4 vignette extremely useful when figuring out how to compute the design matrix for the group-specific effects.

Today, I was adding tests to make sure we are constructing \(\boldsymbol{Z}\) appropriately and found myself comparing the matrices obtained with formulae with matrices obtained with lme4. Then I was like â¦ why not making this a blog post? ð¤

â¦ and so here we are! As a disclaimer, this post mixes both R and Python code. I will try to be explicit when Iâm using one language or the other. But if youâre reading a chunk and it looks like Python, itâs Python, and if it looks like R, itâs R!

Problem

Here we will be comparing design matrices for the group specific terms in a mixed effects model obtained with both lme4 and formulae. Weâre using the dataset Pixel that comes with the R package nlme.

data("Pixel", package = "nlme")
head(Pixel)
## Grouped Data: pixel ~ day | Dog/Side
##   Dog Side day  pixel
## 1   1    R   0 1045.8
## 2   1    R   1 1044.5
## 3   1    R   2 1042.9
## 4   1    R   4 1050.4
## 5   1    R   6 1045.2
## 6   1    R  10 1038.9

Weâre not interested in how to fit a certain model here. Weâre interested in constructing the design matrix for group-specific effects with different characteristics. Weâre using the following formula

f1 = ~ (0 + day | Dog) + (1 | Side / Dog)

where

  • (0 + day | Dog) means that day has a group-specific slope for each Dog. This is usually known as a random slope. The 0 indicates not to add a group-specific intercept (because itâs added later).
  • (1 | Side / Dog) is equivalent to (1 | Side) + (1 | Dog:Side). This means thereâs a varying intercept for each Side and a varying intercept for each combination of Dog and Side. In other words, we have a nested group-specific intercept, where Side is nested within Dog.
lme4_terms = mkReTrms(findbars(f1), model.frame(subbars(f1), data = Pixel))

lme4_terms contains much more information than what we need for this post. We only use lme4_terms$Ztlist, which is a list that contains the transpose of the group-specific effects model matrix, separated by term. These matrices are stored as sparse matrices of dgCMatrix class. If we want to have the sub-matrix for a given group specific term as a base R matrix, we have to do as.matrix(t(lme4_terms$Ztlist$[["term"]])).

names(lme4_terms$Ztlist)
## [1] "1 | Dog:Side"  "0 + day | Dog" "1 | Side"

We have 3 group-specific terms. The first and the last are the group-specific intercepts. These are the result of the nested group-specific intercept (1 | Side / Dog). Dog is nested within Side and consequently thereâs an intercept varying among Side and another varying among Dog within Side. The second term, 0 + day | Dog, represents varying slope of day for each level of Dog.

We finally store the design matrix for each term in different objects that weâll later use when comparing results with those obtained with formulae.

day_by_dog = as.matrix(t(lme4_terms$Ztlist$`0 + day | Dog`))
intercept_by_side = as.matrix(t(lme4_terms$Ztlist$`1 | Side`))
intercept_by_side_dog = as.matrix(t(lme4_terms$Ztlist$`1 | Dog:Side`))

On the other hand, in Python, we use design_matrices() from the formulae library to obtain a DesignMatrices object. All the information associated with the group specific terms is contained in dm.group and the sub-matrix corresponding to a particular term is accessed with dm.group[term_name].

dm = design_matrices("(0 + day | Dog) + (1 | Side / Dog)", r.Pixel)

Thereâs a dictionary called terms_info within dm.group. To access the names of the group-specific effects we just retrieve the keys.

dm.group.terms_info.keys()
## dict_keys(['day|Dog', '1|Side', '1|Side:Dog'])

Names differ a little with the ones from lme4, but they represent the same thing.

day_by_dog = dm.group['day|Dog']
intercept_by_side = dm.group['1|Side']
intercept_by_side_dog = dm.group['1|Side:Dog']

Now letâs compare those matrices!

Each cell in the following plot corresponds to each cell in the matrix. The lowest value for day 0, represented by violet, and the highest value is 21, represented by yellow. The 10 columns represent the ten groups in Dog, and rows represent the observations in Pixel.

In this first case, both panels are representing the same data so we can happily conclude the result obtained with formulae matches the one from lme4. Yay!!

And there it is! Matrices differ because columns are representing different groups. In lme4, groups are looping first along Dog and then along Side, while in formulae it is the other way around.

We can simply re-order the columns of one of the matrices and generate and check whether they match or not.

intercept_by_side_dog_f = as.data.frame(py$intercept_by_side_dog)
colnames(intercept_by_side_dog_f) = py$dm$group$terms_info[["1|Side:Dog"]]$groups
names_lme4_order = paste(
  rep(c("L", "R"), 10), 
  rep(c(1, 10, 2, 3, 4, 5, 6, 7, 8, 9), each = 2), 
  sep = ":"
)

intercept_by_side_dog_f = intercept_by_side_dog_f[names_lme4_order] %>%
  as.matrix() %>%
  unname()

all(intercept_by_side_dog_f == intercept_by_side_dog)
## [1] TRUE

Another formula

This other formula contains an interaction between categoricals as the expression of the group-specific term. In this case, we are going to subset the data so the design matrices are smaller and we can understand whatâs going on with more ease.

# Subset data
Pixel2 = Pixel %>%
  filter(Dog %in% c(1, 2, 3), day %in% c(2, 4, 6)) %>%
  mutate(Dog = forcats::fct_drop(Dog))

# Create terms with lme4
f2 = pixel ~ day +  (0 + Dog:Side | day)
lme4_terms = mkReTrms(findbars(f2), model.frame(subbars(f2), data = Pixel2))
dog_and_side_by_day = as.matrix(t(lme4_terms$Ztlist$`0 + Dog:Side | day`))

And now with design_matrices() in Python.

# Create terms with 
dm = design_matrices("(0 + Dog:Side|day)", r.Pixel2)
dog_and_side_by_day = dm.group["Dog:Side|day"]

dog_and_side_by_day_f = as.data.frame(py$dog_and_side_by_day)
colnames(dog_and_side_by_day_f) = py$dm$group$terms_info[["Dog:Side|day"]]$full_names
side = rep(rep(c("L", "R"), each = 3), 3)
dog = rep(1:3, 6)
day = rep(c("2.0", "4.0", "6.0"), each = 6)
